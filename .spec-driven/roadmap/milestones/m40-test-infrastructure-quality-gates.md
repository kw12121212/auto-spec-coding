# M40 - 测试基础设施与质量门禁

## Goal

在 M39 面向高风险业务回归测试补强的基础上，进一步增强测试基础设施、测试稳定性和统一验证门禁，使新增测试更易编写、更少 flaky，并能为后续变更提供更可信的发布前质量信号。

## In Scope

- 仅增强测试体系本身的可维护性、稳定性和标准化，不以新增单个业务行为回归测试为主要目标
- 抽取已经在多个测试中反复出现且已证明可复用的测试夹具、测试栈、辅助构造器和断言模式
- flaky tests 的根因治理，包括时序脆弱点、共享状态、端口/线程/时间依赖和非确定性输入问题
- 单元测试、集成测试、端到端测试的分层约定与常用命令标准化
- 面向仓库日常开发的 lint、validation、unit/integration test 质量门禁约定补强
- 测试编写与运行相关的最小文档和约定补充

## Out of Scope

- 新增产品功能或改变既有对外行为
- 为单个功能面专门新增大量业务回归测试
- 只为美观或统一形式而进行的测试抽象
- 与当前测试稳定性和质量门禁无关的纯覆盖率堆量工作
- 大规模外部 CI 平台、托管测试云或复杂发布流水线建设
- 为未来假设需求预建过度抽象的测试框架

## Done Criteria

- 系统 MUST 提供可复用的测试夹具或辅助模式，降低新增同类测试的重复样板代码
- 现有高价值测试中的已知 flaky 成因 MUST 被显式修复或隔离，不得继续依赖不稳定时序或共享可变状态
- 仓库 MUST 明确至少一组受支持的 lint 或 validation 命令，以及 unit/integration test 命令，作为测试相关变更的标准验证入口
- 测试分层边界 MUST 更清晰，便于识别某个行为应落在 unit、integration 还是 end-to-end 验证层
- 测试相关文档或约定 MUST 足以指导后续 change 在保持一致风格的前提下扩展测试
- 至少一类仓库级可复用测试模式 MUST 可被后续 proposal 直接复用，而不是仍停留在零散个案中
- 已知 flaky 问题 MUST 优先修根因；若暂时无法根除，隔离策略和边界 MUST 被显式记录

## Planned Changes
- `flaky-test-hardening` - Declared: complete - 修复脆弱时序、共享状态和不稳定等待逻辑，提升测试稳定性和回归信号可信度
- `test-fixture-standardization` - Declared: complete - 统一高频测试夹具、辅助构造器和断言模式，降低新增测试成本并减少重复样板
- `test-command-quality-gates` - Declared: planned - 统一 lint、validation、unit、integration 等常用验证命令与质量门禁约定，形成稳定的测试入口

## Dependencies

- M39 测试补强与回归信心（提供优先补强的高风险测试面）
- M16 集成与发布（提供跨接口一致性与发布验证背景）

## Risks

- 若把质量门禁定义得过重，可能显著增加日常开发反馈时间，削弱实际可用性
- 若测试夹具抽象过度，可能反而掩盖行为语义，让测试更难读和更难诊断失败原因
- flaky tests 若只做表面重试而不修根因，会制造虚假的稳定感

## Status
- Declared: proposed

## Notes

- 本里程碑关注测试体系本身的可维护性与可信度，不替代 M39 对高风险业务回归面的补强工作
- 这里的“质量门禁”强调仓库内受支持的标准验证入口与约定，不默认引入额外外部平台或复杂流水线
- 判定原则：如果目标是让很多测试更容易写、更稳定或更可复用，归入 M40；如果目标是防止某个已有外部行为回归，则应归入 M39
- 若需要执行 Maven 相关验证命令，本里程碑中的测试与验证工作统一使用 `mvnd`，不使用 `mvn`
- 推荐推进顺序：在 M39 的高风险回归测试补强完成后，先执行 `flaky-test-hardening`，再执行 `test-fixture-standardization`，最后执行 `test-command-quality-gates`
