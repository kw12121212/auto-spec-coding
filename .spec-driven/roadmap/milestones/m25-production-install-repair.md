# M25 - 生产环境一键安装与修复

## Goal

提供一个面向受信单机生产环境的运维入口：操作者只需提供目标生产 server 地址，系统即可在受支持的目标环境契约内执行幂等安装或修复、重启服务并完成基础结果校验，用于简化单机生产环境维护，而不是构建 CI/CD 流水线。

## In Scope

- 从脚本所在机器发起到目标生产 server 的一键执行入口
- 明确首期受支持的目标环境契约（远程连接方式、认证方式、操作系统/服务管理器、目录布局）
- 基于 server 地址定位目标环境并建立受支持的远程执行链路
- 幂等安装与修复流程：目标环境缺失组件时安装，已有组件异常时修复
- 服务重启与重启后健康校验
- 结构化执行结果输出，明确安装、修复、重启、校验各阶段结果
- 针对常见可恢复问题的白名单安全修复路径（如目录缺失、配置文件缺失、进程未运行）

## Out of Scope

- CI/CD pipeline、代码提交后自动发布、Git webhook 驱动部署
- 多机器批量发布、滚动发布、蓝绿发布、金丝雀发布
- 云资源创建、DNS/LB/防火墙等基础设施编排
- 复杂配置管理平台或通用运维编排系统
- 对未定义故障的任意远程修复能力
- 超出受支持环境契约的目标机自动兼容

## Done Criteria

- 操作者可提供单个生产 server 地址并触发一次完整的安装或修复流程
- 系统 MUST 明确声明并校验首期受支持的目标环境契约；对不受支持的环境 MUST 快速失败并返回原因
- 当目标机器缺少运行所需产物或目录时，系统可自动补齐并完成安装
- 当目标机器存在已知且在白名单中的可修复环境偏差时，系统可执行修复而不是要求人工先清空环境
- 安装或修复完成后，系统 MUST 执行服务重启
- 重启后系统 MUST 执行至少一项服务可用性校验，并向操作者返回成功或失败原因
- 重复执行同一命令时，流程 MUST 保持幂等，不因已有安装状态而产生破坏性副作用
- 自动化测试 MUST 覆盖受支持环境 happy path、重复执行、已知故障修复、重启失败、健康检查失败和不受支持环境快速失败场景

## Planned Changes

- `production-install-command` - Declared: planned - 定义面向单个生产 server 地址的一键式执行入口、参数契约与结构化结果输出
- `production-repair-workflow` - Declared: planned - 实现生产环境幂等安装与常见可恢复问题修复流程，明确允许自动修复的边界
- `production-restart-verification` - Declared: planned - 实现安装或修复后的服务重启、健康校验与失败诊断输出

## Dependencies

- M16 集成与发布，提供可被安装到目标机器上的稳定产物与使用说明
- M14 HTTP REST API，如健康校验依赖网络接口时需复用现有服务探活能力
- M15 后台进程管理，作为本地或目标端服务生命周期控制行为的参考

## Risks

- 远程执行链路若权限、网络或主机环境前置条件不满足，可能导致流程中途失败
- “自动修复”边界若定义不清，容易演变为不安全的远程运维入口
- 生产环境差异较大时，脚本很容易因为分发路径、服务管理方式或依赖缺失而失效
- 重启成功不等于服务可用，若健康校验过弱会产生误判
- 若不提前锁定受支持环境契约，里程碑很容易持续扩 scope 以适配越来越多的目标机差异

## Status

- Declared: proposed

## Notes

- 该里程碑聚焦“从脚本所在机器发起到单个生产机”的运维闭环，不等同于发布平台建设
- 后续 change 应尽早锁定允许的远程连接方式、目标机前置要求、服务管理器和健康检查标准，避免把实现做成模糊的万能修复脚本
- 若后续需要支持多环境或多主机编排，应另开新 milestone，而不是继续扩张本里程碑范围
- 该里程碑已按产品优先级于 2026-04-17 暂缓；在明确重新排期前，不应作为下一项 roadmap-backed 工作继续推荐
- 当前已按产品优先级暂缓；恢复时应先重新确认首期受支持环境契约，再继续从 `production-install-command` 开始拆分 proposal
